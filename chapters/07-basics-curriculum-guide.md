## 制作物（ToDo管理）について

このカリキュラムでは、各Unitの節目に「制作物：ToDo管理」の章があります。
ここで、そのUnitまでの知識を1つの制作物に統合していきます。

最後は Unit 9 で、同じ題材を Spring Boot のToDo管理API（CRUD API＋DB＋テスト）として完成させます。
さらに Unit 10 で、既存アプリ改修（Petclinic）を通して「現場での改修＋テスト追加」を体験します（おすすめ）。

### 最初の10分：ゴールを言語化する

迷ったら「ToDo管理API（CRUD API＋DB＋テスト）」でOKです。ノート（またはメモ）に次を書き出してください。

1. 作りたいもの
2. 最低限ほしい機能（まずはCRUD API＋DB＋テストまで）
3. 期限（いつまでに動くものにするか）

## AIの使い方（推奨）

このカリキュラムでは、各章の本文コンテキストを持ったAIに質問できます。
「わからない → 止まる」を防ぐために、困ったらすぐに使ってください。

### 使いどころ

- **概念理解**：用語や仕組みを「自分の言葉」で説明できるようにする
- **デバッグ**：エラー文の意味、原因の切り分け、最小修正案
- **レビュー**：コードの読みやすさ、責務分割、命名、重複の指摘
- **理解チェック**：この章の内容で小テストを作ってもらう

### 丸写しを防ぐコツ

- 「答えを丸ごと書かず、まずヒントと確認質問をください」と最初に指定する
- 自分の案（コード/方針）を出してから「レビューして」と頼む

### 1分で書ける質問テンプレ（AI/メンター共通）

質問するときは、次のテンプレを埋めて貼るだけでOKです。
わからない項目があっても、そのままAIに投げて「不足を質問で埋めて」と頼んで構いません。

```text
【状況】（どの章/何をしているか）
【やりたいこと】（最終目的）
【期待する動作】（こうなってほしい）
【実際の動作】（こうなる/エラー）
【エラー全文】（あればそのまま貼る）
【該当コード】（最小の範囲で貼る）
【試したこと】（箇条書き）
【制約】答えを丸ごとではなく、方針→ヒント→確認質問の順で
```

### フィードバックへの修正報告（テンプレ）

PRコメントやレビューで指摘を受けたら、次の形で返すとスムーズです。

```text
【指摘】（引用）
【対応】（どこを/どう直したか）
【確認】（どう動作確認したか）
```

## 課題の提出方法

課題は、理解チェックと練習問題（要件通りにソースコードを書く）の2つがあります。
理解チェックは、各章の学習後に「3分ふり返り」を行ってください。

### 3分ふり返り（理解チェック）

1. 今日の章で新しく出てきた用語を3つ挙げ、1行で説明する
2. 「なぜそうなるか」を1つ説明する（例：型変換、参照、例外など）
3. 今日の要点を2〜3行でまとめる

要件通りにソースコードを書く課題は、原則 GitHub にpushしてリポジトリURLで提出してください（差分が追いやすく、AIを使った変更の確認もしやすいです）。
どうしても難しい場合のみ、zipにまとめて提出してもOKです。

何か疑問やわからない点があれば、遠慮せずにお知らせください。
確認次第、迅速にサポートいたします。

## Javaプロジェクトの作り方

1. IDEを開く（Eclipse / IntelliJ IDEA / VS Code など）
2. Javaプロジェクトを作成する（例: `java-practice`）
3. パッケージを作成する（例: `com.example.practice`）
4. クラスを作成する（例: `Main`）
5. `main` メソッドを用意して `Hello World` を表示する
6. IDEの実行機能で実行する
7. コンソールに `Hello World` が表示されることを確認する
