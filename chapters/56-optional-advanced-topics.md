# 補講（任意）：発展トピックをAIで学ぶ

この章は必須ではありません。
Unit 9（Spring BootのToDo管理API）と Unit 10（Petclinic改修）が終わってから、必要になったときに読みましょう。

## 方針（AI時代の学び方）

- 覚えるより「必要になったら調べて使う」を優先する
- 調べるときは、**自分の目的** と **自分のコード/エラー** をセットでAIに渡す
- 答えの丸写しではなく「方針→ヒント→確認質問」の順で引き出す

## 発展トピック（例）

次のテーマは、最初から全部やらなくてOKです。
制作物（ToDo API / Petclinic改修）で必要になったタイミングで触れましょう。

- **設計**：SOLID、責務分割、DTO/Entityの分離、例外設計、依存性注入（DI）
- **テスト**：Unit/Integrationの使い分け、テストデータ設計、カバレッジの見方
- **性能**：N+1問題、インデックス、キャッシュ、ログでのボトルネック特定
- **並列/非同期**：スレッド安全性、排他制御、Executor、非同期処理の落とし穴
- **セキュリティ**：入力検証、SQLインジェクション、認可、秘密情報の扱い
- **運用**：ログ設計、設定の外出し、デプロイ、監視（メトリクス）

## 進め方（おすすめ）

1. 「いま困っていること」を1行で書く（例：N+1が出た、テストが書けない）
2. 期待する動作/状況/エラーを貼る（必要なら関連コードも貼る）
3. AIに「原因候補→確認手順→最小修正案」の順で出してもらう
4. 自分で直したら、AIにレビューさせる（危険箇所だけ指摘してもらう）

## AIに質問する（この章の例）

次の例をそのままAIに投げてOKです（必要なら自分のコード/エラーに置き換えてください）。

- 「このToDo APIの設計をレビューして。責務分割（Controller/Service/Repository/DTO/Entity）の危険箇所だけ指摘して、直し方はヒントで」
- 「UnitテストとIntegrationテストの使い分けを、このプロジェクト（ToDo API / Petclinic）に当てはめて提案して」
- 「性能が不安。N+1が起きる典型パターンと、Spring Data JPAでの確認方法を手順で教えて」
- 「並列処理が必要になった。スレッド安全性のチェック観点を、このコード（貼る）でレビューして」
